深入理解Java虚拟机

JAVA
JAVA • JVM

基础概念

  • java方法：正常的java方法

  • native方法，即本地方法：是java调用非java代码（如C++代码）的接口，使用native方法可以用来使用底层的主机平台的某个特性，加快访问老系统或者已有的库，以及加快程序运行的速度

  • JDK： Java SE Development Kit（Java开发工具），提供了编译和运行Java程序所需的各种资源和工具，包括：JRE+java开发工具。

  • JRE： Java runtime environment（Java运行环境），包括：虚拟机+java的核心类库。

  • JVM： Java核心虚拟机，是一个可以执行Java字节码的虚拟机进程

  • hotspot VM：热点代码技术，是现在官方JDK采用的VM，也是使用最多的。

  • OOM：内存溢出异常。

  • JIT：即时编译器，一般情况下可以提高java程序编译速度。

2.2 内存区域具体分析

  • 程序计数器PCR

    • 对于native方法：undefined，为空
    • 对于java方法：记录虚拟机正在执行的字节码指令地址
      • 类似于计算机内部的PC，用于指向当前执行代码行，并通过修改指定下一条执行代码行
      • 不会出现OOM
    • 每个线程都需要一个独立的程序计数器，故线程私有

  • 虚拟机栈

    • 即所说的java栈内存，其生命周期与线程相同。

    • 服务于java方法（即字节码），描述的是Java方法执行的内存模型：

      • 每个方法被执行时都会同时创建一个栈帧用于存储局部变量表、动态链接、方法出口等信息。
        • 栈帧是java方法运行期的基础数据结构
      • 每一个方法被调用直至完成的过程，对应着一个栈帧在虚拟机栈中从入栈到出栈的过程。

    • 可能抛出异常：

      • OutOfMemoryError（在虚拟机栈可以动态扩展的情况下，扩展时无法申请到足够的内存）
      • StackOverflowError（线程请求的栈深度 > 虚拟机所允许的深度）

    • 线程私有

  • 本地方法栈

    • 类似虚拟机栈，与之区别在于：本地方法栈为native方法服务
    • 线程私有

  • 堆

    • 即堆内存，唯一目的是存放对象实例
    • JVM启动时创建，是垃圾收集器管理的主要区域，故又叫做“GC堆(Garbage Collected Heap)”。
    • 可以处于物理不连续但逻辑连续的内存空间中。
    • 可能抛出OOM异常
    • 线程共享

  • 方法区

    • 存储已被JVM加载的类信息，常量，编译后的代码等，垃圾收集很少在此进行，但若经常动态生成Class的应用（如spring），则需要经常回收
    • 组成部分有：运行时常量池
      • 运行时常量池：各类符号引用，字面量（即直接量、给定的右值）等经过类加载后存储到此。具有动态性，运行时可放入
    • 可能抛出OOM异常
    • 线程共享

  • 直接内存

    • 不属于虚拟机运行时数据区，也不是JVM内存区域，但这部分被频繁使用，并且也可能抛出OOM

    • 在JDK1.4增加了NIO，可以分配堆外内存（系统内存替代用户内存），避免Java堆和native堆频繁切换，提高了性能

2.3.2 对象的结构（内存布局）

  • 对象头：
    • 第一部分：存储对象自身运行时的数据，HashCode、GC分代年龄等（Mark Word）；
    • 第二部分：类型指针，指向它的类元数据的指针，虚拟机通过这个指针来判断这个对象是哪个类的实例（HotSpot 采用的是直接指针的方式访问对象的）；
    • 如果是个数组对象，对象头中还有一块用于记录数组长度的数据。
  • 实例数据：
    • 默认分配顺序：longs/doubles、ints、shorts/chars、bytes/booleans、oops (Ordinary Object Pointers)，相同宽度的字段会被分配在一起，除了 oops，其他的长度由长到短；
    • 默认分配顺序下，父类字段会被分配在子类字段前面。

2.3.3 对象访问的两种方式

  • 句柄访问

    • 采用这种方式，Java堆中会划分出一块作为句柄池。reference->句柄，句柄保存了对象实例数据和类型数据等信息的地址。
    • 好处：句柄地址稳定，对象被移动时只需要改变句柄中的实例数据指针。

  • 直接指针访问

    • 这种方式，reference直接存储对象地址。
    • 好处：速度快，HotSpot采用这种方式。

2.4 OOM内存溢出异常

Java堆溢出

  • 产生原因：不断创建对象，同时保证GC Roots到对象之间有可达路径来避免垃圾回收，这样最终就会产生OOM
  • 异常堆栈信息：java.lang.OutOfMemoryError: Java heap space
  • 解决方式：
    1. 首先通过内存映像分析工具对dump出来的堆转储快照分析，确认内存中对象是否必要，也分清楚是内存泄露还是内存溢出。
    2. 内存泄露：查看泄露对象到GC Roots的引用链，定位泄露的位置
    3. 内存溢出：检查虚拟机堆参数，适当调整。代码上则尝试减少程序运行期的内存消耗

Java虚拟机栈和本地方法栈溢出

  • 产生原因：
    • 线程请求栈深度大于虚拟机所允许的最大深度–栈溢出
    • 虚拟机在扩展栈时无法申请到足够内存–内存溢出
  • 异常堆栈信息：StackOverflowError 以及 OutOfMemoryError
  • 解决方式：
    • 单线程下，栈帧过大、虚拟机容量过小都不会引起内存溢出，只会栈溢出。
    • 多线程下，由于每个新线程占用一定内存，当线程数太多时，可能出现内存溢出。解决方式是减少最大堆和栈容量来换取更多的线程。

方法区和运行时常量池溢出

  • 产生原因：如用String.intern()这个Native方法不断给常量池添加某字符串值，或者产生大量的类到方法区，就可能产生溢出。
  • 异常堆栈信息：java.lang.OutOfMemoryError: PermGen space
  • 由于垃圾回收器一般很少在此回收，所以在经常产生大量Class的应用中，需要特别注意类的回收情况。

本机内存直接溢出

  • 异常堆栈信息：Heap Dump 文件中看不见明显异常，程序中直接或间接用了 NIO

3.2 如何回收：垃圾收集算法

​

  • minor GC：从年轻代空间（包括 Eden 和 Survivor 区域）回收内存。速度快，频率高
  • major GC：清理老年代
  • full GC：清理整个堆空间，包括年轻代和老年代

3.3 HotSpot中实现GC

  • 主要流程是：枚举根节点 —> 找到要清除的对象 —> 清除对象
  • 枚举根节点：
    • 通过查询OopMap上的引用关系来枚举
    • 然而为了保持一致性，GC时需要暂停所有Java线程，即Stop the world
    • 何时暂停？在安全点暂停
  • 安全点：
    • 能够保持程序长时间执行的地方，如循环跳转等，此处进行GC
    • stop the world方法：抢先式中断和主动式中断（更常用）
  • 提供安全区域：
    • 此代码片段不再有引用关系的变化，故此区域任意地方开始的GC都是安全的

3.4 HotSpot中的7个垃圾收集器

  • Serial收集器

    • 这个收集器是一个单线程的收集器，它会以一条收集线程去完成垃圾收集工作，并且在进行垃圾收集时，必须暂停其他所有的工作线程，直到它收集结束

  • ParNew

    • Serial的多线程版本

  • Parallel Scavenge

    • 是一个新生代收集器，它是使用复制算法的收集器，又是并行的多线程收集器

  • Serial Old

    • 是Serial收集器的老年代版本,是一个单线程收集器，使用标记整理算法

  • Parallel Old

    • Paraller Seavenge收集器的老年代版本，使用多线程和标记整理算法

  • CMS

    • CMS收集器是基于标记清除算法实现的，

    • 流程：初始标记 —> 并发标记 —> 重新标记 —> 并发清除

    • 优点：并发收集、低停顿

    • 缺点：

      • CMS收集器对CPU资源非常敏感，CMS默认启动的回收线程数是$\frac{CPU数量+3}{4}$

      • CMS收集器无法处理浮动垃圾，可能出现Failure失败而导致一次Full GC 产生

      • CMS是基于标记清除算法实现的，效率不够高

  • G1

    • 优点
      • 并行与并发：利用多CPU缩短STOP-The-World停顿的时间
      • 分代收集
      • 空间整合：不会产生内存碎片
      • 可预测的停顿
    • 流程：初始标记 —> 并发标记 —> 最终标记 —> 筛选回收

C6 类文件结构

6.1 无关性的基石

  • 实现平台无关性的基石是字节码。

  • 实现语言无关性的基础是虚拟机和字节码存储格式。编译器将代码编译成存储字节码的Class文件，虚拟机并不关心Class的来源，只要符合结构就可在JVM中运行。

6.2 Class类文件的结构

  • Class文件格式如下（u4表示4个字节无符号数，下列数据项先后顺序、长度、含义都不能改变）：

  • 前4个字节：魔数
    • 唯一作用是确定该文件是否是一个能被JVM接受的Class文件。0xCAFEBABE
  • 5~8字节：Class文件版本号
    • 5-6字节是次版本号(如JDK 1.8.X中的X)，7-8字节是主版本号（如JDK 1.8）
  • 接下去是常量池，数量不固定，起始2字节表示常量池的容量计数值，后面的是具体常量
    • 但注意与寻常不同的是，容量计数值从1开始，若为0表示不引用任何一个常量池项目
    • 常量池存放字面量和符号引用
      • 字面量：文本字符串，被声明为final的常量值等
      • 符号引用：类和接口的全限定类名，字段或方法的名称和描述符
    • 具体常量的类型是cp_info，即每项常量都是一个表，共有11种不同的表结构数据，每种表数据结构开始第一位是tag标志位，取值为$[1,2)\bigcap(2,12] $，后面的根据种类而不同
  • 接下去是2个字节的访问标志，用于识别一些类或接口层次的访问信息，如：是接口还是类，是否public，是否抽象类，是否final等
  • 接下去是类索引，父类索引，接口索引集合
    • 类索引：类的全限定名
    • 父类索引：父类的全限定名，由于Java不允许多重继承，故只有一个。且因为Object是所有类的父类，故除了Object外该项数据都不为0
    • 接口索引：表示实现的接口集合（implements或extends（本身是接口））
    • 索引访问时，先指向寻找常量池相关表数据结构，再指向常量池utf8_info中的全限定类名字符串
  • 字段表：用于描述接口或类中声明的变量，包括作用域，类级还是实例级（static），可变性（final），并发可见性（volatile），基本数据类型等
    • 描述时也要通过引用常量池和访问标志来描述
  • 方法表：类似字段表，只包括在该类中显式定义的方法
  • 属性表：在Class文件、字段表、方法表中都可以携带自己的属性表集合，来描述某些场景专有信息

C7 虚拟机的类加载机制

7.1 类加载的时机

  1. 类加载机制：虚拟机把描述类的数据从Class文件中加载到内存，并进行数据校验，转换解析和初始化，形成可以被JVM直接使用的Java类型
  2. Java动态扩展的语言特性：运行时动态加载和连接
  3. 类加载的生命周期：加载 –> 验证 –> 准备 –> 解析 –> 初始化 –> 使用 –> 卸载。其中验证、准备、解析三步合起来是连接的过程
  4. 何时开始加载？
    • 主动引用：当且仅当遇到下列情况才会“初始化”，自然也需要加载了
      • 遇到 new、getstatic、putstatic、invokestatic 字节码指令。后三个字节码指令是指读取或设置一个类的静态字段
      • 对类进行反射调用时，若类没有初始化，就需要初始化
      • 初始化一个类时，若其父类未初始化，就需要初始化
        • 但处理接口时，其父类是需要使用时才初始化
      • 包含main()方法的那个主类
    • 被动引用：不会主动初始化
      • 通过子类引用父类静态字段，不会导致子类初始化
      • Array[] arr = new Array[10] 不会触发 Array 类初始化
      • static final VAR 在编译阶段会存入调用类的常量池，通过 ClassName.VAR 引用不会触发 ClassName 初始化

7.2 类加载的过程

  1. 加载：查找并加载类的二进制数据
    • 完成下列3件事情：
      • 通过一个类的全限定类名来获取定义此类的二进制字节流
      • 将这个字节流所代表的静态存储结构转换成方法区的运行时数据结构
      • 在Java堆中生成一个这个类的Class文件，作为方法区这些数据的访问入口
        • hotspot会放在方法区，而不是Java堆
    • 显式加载
      • ClassLoader.loadClass(className)或Class.forName(className)
      • 两者区别
        • 使用ClassLoader.loadClass()来加载类，不会执行初始化块，而后者会
        • Class.forName 借助当前调用者的 class 的 ClassLoader 完成 class 的加载。故传参时若传入(false, loader)也可以不执行静态块
    • 隐式加载
      • new，使用类的静态域，创建子类对象，使用子类的静态域，其他的在JVM启动时的隐式加载。
  2. 验证：确保被加载类的准确性，此步骤重要但非必须
    • 进行4项验证
      • 文件格式验证：检查魔数，版本号，常量池中常量类型等
      • 元数据验证：语义分析，判断是否符合Java语言规范。如是否有父类等
      • 字节码验证：程序语义是否合法，符合逻辑
      • 符号引用验证：对类自身以外的信息（常量池中的符号引用）进行匹配性校验
  3. 准备：为类的静态变量分配内存，并初始化为默认值
    • 这时候进行内存分配的仅包括类静态变量（static），而不包括实例变量，实例变量会在对象实例化时随着对象一块分配在Java堆中
    • 这里的初始化仅仅是默认值，而非真正的赋值。真正赋值在初始化阶段完成
      • static变量和全局变量不显式赋值则会给默认值。局部变量一定要显式赋值
      • 对于static final int a = 3这样的，则这一步就初始化a=3，因为后面不会再修改。即被动引用的第三种情况
      • reference类型若没有显式赋值，则默认为null
  4. 解析：把类中的符号引用转换为直接引用
    • 这里的直接引用指：直接指向目标的指针、相对偏移量或一个间接定位到目标的句柄
    • 这里的符号引用指：以一组符号描述所引用的对象（如对象的全类名），引用的目标不一定存在于内存中
    • 在这一步之前，常量池中的引用是不一定存在的，解析过之后，可以保证常量池中的引用在内存中一定存在。
  5. 初始化：为类的静态变量赋予正确的初始值，即执行类构造器 <clinit>() 方法的过程
    • <clinit>()的任务：
      • 执行所有 static 的赋值操作
      • 执行static 块中的语句
    • <clinit>()的细节：
      • <clinit>()是由编译器自动收集类中的静态赋值和静态代码块，收集顺序按源文件顺序。且静态语句块只能访问定义在它前面的变量，定义在它后面的变量，只可以赋值，不能访问
      • <clinit>()与对象实例构造器<init>()不同，它不需要显式地调用父类构造器。且<init>()是在创建对象实例时才会调用。
      • 由于父类的<clinit>()，故父类中定义的静态语句块要优于子类中的变量赋值
    • 接口与类处理方式区别：
      • 对于接口的<clinit>()不会先执行父类的<clinit>()
    • 关于锁
      • 同一时刻只能有一个线程在执行<clinit>() 方法，因为虚拟机要保证在同一个类加载器下，一个类只被加载一次
    • 若没有静态赋值和静态代码块，初始化步骤可以省略

7.3 类加载器

  1. 判断两个类相等：

    • 方法：instanceof、equals、isInstance、isAssignableFrom
    • 要求：同一个.class文件，被同一个虚拟机加载，被同一个类加载器加载

  2. 双亲委派模型：

    • 系统提供的3种类加载器：

      • 启动类加载器（Bootstrap），负责加载<JAVA_HOME>/lib或者-Xbootclasspath 参数指定的路径
      • 扩展类加载器（Extension），负责加载<JAVA_HOME>/lib/ext或者java.ext.dirs 系统变量指定的路径
      • 应用程序加载器（Application），负责加载-classpath 参数

    • 双亲委派模型

      • 流程：即类加载器在接收到类加载请求时，首先将加载任务交给父类加载器，递归向上直至顶层启动类加载器。若启动类加载器加载失败，则再向下交付。最后父类加载器都加载失败，才自己加载
      • 实现：先检查是否加载过，若没有加载就调用父加载器loadClass方法。若父加载器为空则默认用启动类加载器作为父加载器。若父加载器最终返回加载失败，则在抛出ClassNotFoundException异常后，再调用自己的findClass方法进行加载。
      • java.lang.ClassLoader 的 loadClass() 实现了双亲委派模型的逻辑，因此自定义类加载器一般不去重写它，但是需要重写 findClass() 方法
      • 好处：
        • 让Java类同其类加载器一起具备了一种带优先级。比如我们要加载java.lang.Object类，无论我们用哪个类加载器去加载Object类，这个加载请求最终都会委托给Bootstrap ClassLoader，这样就保证了所有加载器加载的Object类都是同一个类。这样保证了Java程序运行的稳定性。

C8 虚拟机字节码执行引擎

8.1 运行时栈帧结构

  1. 栈帧：用于支持虚拟机进行方法调用和方法执行的数据结构，是虚拟机运行时数据区的虚拟机栈中的栈元素。 可以理解为一个进入栈中的方法。
  2. 栈帧包括：局部变量表，操作数栈，动态连接和方法返回地址
    • 局部变量表：存放方法参数和方法内部定义的局部变量，局部变量表容量用slot为最小单位。
      • 一个slot可以存放一个32位以内（boolean、byte、char、short、int、float、reference和returnAddress）的数据类型。
      • 64位用两个slot，是线程安全的
      • Slot复用：如果PCR当前位置已经超过某个变量的作用域时，例如出了定义这个变量的代码块，这个变量对应的 Slot 就可以给其他变量使用了。
        • Slot影响gc：关键在于局部变量表中的 Slot 是否还存在关于 某变量placeholder 的引用
        • 也就是说，仅仅出了placeholder的作用域，就进行GC，placeholder不一定会被回收掉
        • 此时若有了int a = 0类似语句，占用了原来placeholder的slot，再GC，placeholder才会被回收掉
    • 操作数栈：起初为空，执行过程中不断入栈/出栈
    • 动态连接：指向运行时常量池中该栈帧所属方法的引用
    • 方法返回地址
      • 退出方法的方式：return或者异常
      • 退出的操作：恢复现场，将返回值压入操作数栈以便后续取出查看，修改PC指向下条指令

8.2 方法调用

  1. 概述：方法调用仅仅是确定被调用的方法的版本，执行方法是基于栈的字节码解释执行的过程。Class文件的编译过程不包含传统的连接步骤，一切方法在Class文件中存储的都是符号引用，而非实际入口地址。

  2. 解析：有一部分方法在类加载的解析步骤可以转换为直接引用，但需要满足“编译器可知，运行期不变”

    • 这些方法不能通过继承或别的方式重写，不会被覆盖，故要可在运行前确定调用版本

    • 符合这个要求的有：

      • 静态方法，通过invokestatic调用
      • 私有方法，构造器方法，父类方法，通过invokespecial调用

    • Java虚拟机提供的4条方法调用字节码指令

      • invokestatic：调用静态方法
      • invokespecial：调用实例构造器方法，私有方法，父类方法
      • invokevirtual：调用所有虚方法（不包括final）
      • invokeinterface：调用接口方法，会在运行时再确定一个实现此接口的对象

    • 有invokestatic和invokespecial可以在解析阶段确定唯一的调用版本，而final修饰的方法虽然是invokevirtual，但也不能通过继承或别的方式重写，故也属于非虚方法。

  3. 分派：针对多态（重载，重写）

    • Java面向对象三个基本特征：封装，继承，多态。其中多态重要特征是：
      • 重载：发生在同一个类中，入参必须不同，返回类型、访问修饰符、抛出的异常都可以不同
      • 重写：发生在子父类中，入参和返回类型必须相同，访问修饰符大于等于被重写的方法，不能抛出新的异常
    • 静态分派：方法重载
      • Human man = new Man()，其中Man extends Human，那么Human是变量的静态类型，Man是变量的实际类型。
        • 接着执行man=new Women()，这是发生了实际类型的改变
        • 若执行(Women)man，这是发生了静态类型的改变
      • 在重载时，编译器是通过方法参数的静态类型（类型，个数，顺序等），而不是实际类型，来判断调用哪个方法
      • 另外是静态分派的顺序问题，如传递一个字符’a'，它可以依次作为char，int（unicode数值），long，序列化，Object等。当然实际情况重载一般不会发生这种情况
    • 动态分派：方法重写
      • 先找本身，本身没有再找父类，参数类型匹配看实际类型
      • 实现动态分派常常使用虚方法表进行优化，提高分派性能和效率
        • 在类的方法区建立虚方法表，虚方法表中存放着各个方法的实际入口地址，若某个方法没有被子类方法重写，那子类方法表中该方法的入口地址即父类方法表中该方法的入口地址
        • 使用这个方法表索引代替在元数据中查找，该方法表会在类加载的连接阶段初始化好
    • 单分派：根据方法接收者的实际类型进行分派，Java动态分派属于单分派
    • 多分派：根据方法接收者的静态类型和方法参数类型进行分派，Java静态分派属于多分派

8.3 基于栈的字节码解释执行引擎

  • 栈即操作数栈，整体流程就是通过javac将程序编译成字节码，JVM加载字节码，解释执行字节码生成机器码
  • 解释的过程是：词法分析 -> 语法分析 -> 形成抽象语法树 -> 遍历语法树生成线性字节码指令流
  • 而具体执行过程是通过基于栈的指令集或者基于寄存器的指令集完成的。整体过程类似汇编语言的指令集

8.4 语言类型支持

  • 概念：类型检查的主体过程在运行期，而非编译期的编程语言

  • 优点

    • 静态类型语言：编译器提供了严谨的类型检查，类型相关的问题能在编码的时候就发现
    • 动态类型语言：灵活性和效率高

  • 上图的强类型指不经过强制类型转换，变量类型不会改变。弱类型则相反

  • Java属于静态类型语言，但JVM提供了动态类型支持

    • invokedynamic：如lambda表达式
    • java.lang.invoke：如methodHandler这种动态确定目标方法的调用机制

C9 编译期优化

9.1 概述

  • 三类编译器：
    • 前端编译器：如Javac，把*.java 文件转变成 *.class文件的过程
    • JIT编译器：后端运行期编译器把字节码转变成机器码的过程
    • AOT编译器：静态提前编译器
  • Java中即时编译器在运行期的优化对程序运行很重要，因为可以让那些不是由Javac生成的Class文件（如Groovy，JRuby等产生的）也能被优化编译。而Javac在编译器的优化过程对于程序编码有好处。

9.2 Javac编译器

  • 编译的三个过程：解析与填充符号表，插入式注解处理器注解处理过程，语义分析与字节生成过程
  • 解析与填充符号表：解析（包括编译原理中的词法分析和语法分析）+ 填充符号表
    • 词法分析：将源代码字符流转变为Token集合
    • 语法分析：根据Token序列来构造抽象语法树(AST)
    • 填充符号表：生成符号地址和符号信息的键值对
  • 注解处理器：JDK1.5后提供，注解与普通Java代码一样，在运行期发挥作用
  • 语义分析与字节码生成：
    • 语义分析：对结构上正确的源程序进行上下文有关性质的审查，如类型审查
      • 标注检查：变量使用前是否被声明，变量与赋值之间的类型匹配，常量折叠等
      • 数据及控制流分析：局部变量使用前是否赋值，方法每条路径是否有返回，受查异常的处理等
    • 解语法糖：Java是低糖语法，语法糖包括：泛型，变长参数，自动装箱拆箱
    • 字节码生成：前面步骤生成信息转化成字节码，同时进行少量代码添加与转换工作
      • 如<init>和<clinit>就是这个阶段被添加到语法树中的

10.1 即时编译器：JIT编译器

      • 常见编译性语言：C，C++。

10.3 热点代码编译

  • 热点代码：被多次调用的方法，或者被多次调用的循环体所在的方法

  • 如何判定是热点代码：

    • 使用基于采样的热点探测：如果一个方法经常在栈顶，则认为是热点代码。这样的优点是容易获取方法调用关系，简单高效，但是精准度不够，容易受到线程阻塞的影响
    • 使用基于计数器的热点探测：每个方法计数，超过一定值认为是热点代码。这样精确严谨，但是实现复杂

  • Hotspot中的两种计数器

    • 方法调用计数器：统计方法调用的次数
    • 回边计数器：统计方法回边的次数，即方法体重循环体代码执行的次数。回边是指在字节码中遇到控制流向后跳转的指令

10.4 经典优化技术

  • 公共子表达式消除–语言无关

  • 数组范围内检查消除–语言相关

    • 循环时，若可以判断循环变量在数组索引范围内，则循环内部使用循环变量作为数组索引，可以不进行越界检查。

  • 方法内联

    • 目的：为了减少方法调用的开销
    • 主要过程就是把多个方法”复制“到一个方法中，作为内联而非调用，减少开销
    • 对于非虚方法，直接内联
    • 对于虚方法，
      • 只有一个目标版本：直接内联，但这是激进优化，要给出逃生门
      • 多个目标版本：
        • 内联缓存。一开始为空，第一次调用后缓存内记录调用方法的对象版本信息。
        • 以后每次调用都与第一次比较。
        • 一致则继续使用内联。
        • 不一致则说明多态、取消内联、查找虚方法进行方法指派

  • 逃逸分析

    • 主要过程：分析对象的作用域，查看是否在作用域之外使用

      • 方法逃逸：对象在方法中定义之后，能被外部方法引用，如作为参数传递到了其他方法中
      • 线程逃逸：对象在方法中定义之后，能被外部方法引用，如作为参数传递到了其他方法中

    • 对于不会逃逸到方法或者线程外的对象能进行优化

      • 栈上分配：在栈上分配内存，故这个对象会因为出栈而自动销毁，减少垃圾收集的开销
      • 标量替换：
        • 标量：基本数据类型和reference
        • 具体实现：不创建对象，而将对象分为数个标量，在栈上分配
      • 锁消除：不会逃逸到线程外的方法不进行同步

C11&12 Java内存模型与线程、锁

C11 内存模型与线程

1) 并发编程的三大性质

  • 原子性：一个整体，要么全部执行，要么全部不执行，不会被中断
  • 可见性：当多个线程访问同一个变量时，一个线程修改了这个变量的值，其他线程能够立即看到修改的值
  • 有序性：即程序执行的顺序按照代码的先后顺序执行

2) 内存间操作流程

  • 这与处理机–高速缓存–主内存的关系尽管并不是一个层次，但对应关系是一致的

3) 原子操作

操作 功能 作用于
lock 把一个变量变为线程独占 主内存
unlock 释放一个被锁定的变量 主内存
read 把一个变量的值从主内存转移到工作内存 主内存
load 把read操作得到的值放入工作内存的变量副本中 工作内存
use 把工作内存中一个变量的值交给执行引擎 工作内存
assign 把一个从执行引擎接收的值给工作内存的变量 工作内存
store 把工作内存一个值传递给主存 工作内存
write 把store得到的变量放入主存的变量中 主内存

4) Happen-Before规则

  • 程序次序规则：代码执行顺序规则
  • 管程锁定规则：同一个锁，unlock操作先行于lock操作
  • volatile变量规则：volatile变量的写操作先于读操作
  • 线程启动规则：线程的start方法先于此线程其他操作
  • 线程终止规则：线程其他所有操作先于对该线程的终止检测（如join方法，isAlive方法）
  • 线程中断规则：对线程interrupt方法调用先于被中断线程的代码检测到中断事件的发生（isInterrupted方法）
  • 对象终结规则：一个对象的初始化完成先于其finalize方法的开始
  • 传递性：先行性具有传递性

5) volatile变量的特殊规则

  • 可见性的实现常常通过主内存作为中间媒介的
  • volatile变量不存在一致性问题，因为每次读取都要刷新以下。但是volatile变量的运算在并发下不一定是安全的，这是因为Java里面的运算并非原子操作。即volatile能保证一致性和有序性，不能保证原子性
  • volatile使用的规则（下面记V,W是两个volatile变量）
    • 在工作内存中，每次使用V前都必须先从主内存刷新最新的值
    • 在工作内存中，每次修改V后都必须立刻同步回主内存中
    • volatile修饰的变量不会被指令重排序

C12 锁相关

1)概念

  • 使用线程安全的容器的线程未必一定是安全的，很多时候外围还需要加上内置锁

  • 线程安全的实现方法

    • 互斥同步：如synchronized，ReentrantLock
      • ReentrantLock相比synchronized，前者是通过lock和unlock，后者是原生语法层面
      • 前者多了：等待可中断（长久不释放锁时，等待线程可以放弃等待，做其他事）、公平锁（按照申请锁的时间顺序获得锁）、锁绑定多个条件
    • 非阻塞同步：如testandset，CAS，incrementandget
    • 无同步方案：有一些代码天生线程安全
      • 可重入代码：随时中断，相同数据一定返回相同结果
      • 线程本地存储：threadlocal

  • CAS（比较相同再交换）是一种乐观锁、是一种原子操作

    • 须有三个操作数：内存中的值V，旧预期值A、要修改的新值B。
    • CAS执行：当且仅当V符合A时，处理机用B更新V，否则就不执行更新。但无论如何，都会返回V的旧值
    • CAS的漏洞：ABA问题，即一个变量先从A变成了B又变回A，CAS认为没变

  • CAS的实现：例如Java的AtomicInteger调用incrementAndGet，这个类内部包含unsafe类，这个unsafe类中提供了CAS的原子操作。

    • 这个unsafe类可以让Java也使用指针，当然这很危险，所以只能通过反射的方式来获得这个类

    • unsafe类：

      • 一个volatile修饰的value，用来保存这个数字值

      • 一个offset，用来找到value的内存地址

      • 源码：

      private volatile int value;
public final int incrementAndGet() {
        return U.getAndAddInt(this, VALUE, 1) + 1;
}
    
public final int getAndAddInt(Object o, long offset, int delta) {
        int v;
        do {
            // v是旧的预估值
            v = getIntVolatile(o, offset);
            // o是内存最新值，v是旧的预估值
        } while (!weakCompareAndSetInt(o, offset, v, v + delta));
        return v;
}

    • v变量是旧的预估值，先通过getIntVolatile找到预估值，此处一开始就是value中的值

    • 假如这个线程t1暂停，t2也走到do while循环处

    • 此时两个线程竞争，假如t1先走，调用while内的语句，这个语句参数 o是内存最新值，v是旧的预估值，v+delta就是新的值，于是这里t1有o==v，所以v+delta能成功写入，返回true，取非，于是退出

    • 而此时t2再走，发现旧的预估值和内存最新值不相等，于是会重新执行循环，直到成功写入

2)锁优化

  • 自旋锁：互斥同步对性能影响最大的是阻塞的实现，挂起和恢复都需要在内核态完成，开销大。所以引入自旋锁：
    • 一个线程在等待另一个运行线程释放锁时，不去挂起和恢复，而是执行一个忙循环（自旋），这样减少开销，提高并发度
    • jdk1.6以后有了自适应的自旋锁：JVM会根据以往自旋的时间，来决定是分配更长自旋时间还是更短。即JVM对程序锁的状况预测的越来越准确
  • 锁消除：虚拟机在即时编译过程中，对一些代码上要求同步，但是被检测到不可能存在共享数据竞争的锁进行消除。依据是逃逸分析的数据支持
  • 锁粗化：如果一系列的连续操作都对同一个对象反复加锁和解锁，就可以把锁范围扩展到整个操作序列的外部，减少加解锁次数
  • 轻量级锁：相对于系统互斥量的传统锁而言的轻量级。是为了减少传统重量级锁的性能消耗
    • hotspot虚拟机对象头markword中存放了关于锁的相关信息
    • 执行过程：
      • 在代码进入同步块的时候，如果该对象没有被锁定（锁标志位为01），就在栈帧中建立一个名为锁记录的空间，用于存储锁对象目前的markword拷贝页。
      • 然后虚拟机使用CAS将对象的markword更新为指向LOCK Record的指针。
      • 如果这个更新操作成功了，这个线程就拥有了该对象的锁，并且对象markword的锁标志位变为00，即处于轻量级锁定的状态
      • 如果更新失败，虚拟机首先检查markword是否指向当前线程的栈帧，如果是就说明当前线程已经拥有了这个对象的锁，可以直接进入同步代码块执行，否则说明这个锁被其他线程抢占了
      • 当有两条以上线程争夺一个锁时，这个轻量级锁就膨胀为重量级锁，锁状态变为10，markword中存储的就是重量级锁
    • 获取锁的总结：栈帧放markword来获取锁 –> 修改标志位，CAS把锁id记录到栈帧中markword –> ….
    • 解锁：同样通过CAS
      • 使用CAS将Mark Word替换回到对象头；
      • 如果成功，则表示没有竞争发生；
      • 如果失败，则表示当前锁存在竞争，锁就会膨胀为重量级锁。
  • 偏向锁
    • 目的：为了消除在无竞争状态下的同步原语，提高性能
    • 轻量级锁是在无竞争状态下使用CAS去消除互斥量，偏向锁是无竞争状态下消除整个同步
    • 偏向：偏向于第一个获得它的线程，如果在接下来的执行中，该锁没有被其他线程获取，则持有偏向锁的线程将永远不需要再进行同步
    • 执行：
      • 先像轻量级锁一样获取锁，当锁对象第一次被获取后，JVM修改对象头标志位为01，即偏向模式
      • 同时使用CAS把这个锁的id记录到对象markword中
      • 若操作成功，这个有偏向锁的线程以后进入这块代码不再需要任何同步操作
      • 当另一个线程试图去获取这个锁时，偏向模式就结束了
      • 根据对象目前是否被锁定，恢复到未锁定和轻量级锁定状态
    • 获取锁的总结：栈帧放markword来获取锁 –> 修改标志位，CAS把锁id记录到栈帧中markword –> ….

专题1：synchronized锁升级案例

  1. 锁升级过程：

    • 在jdk1.5之前，synchronized只包含monitor这个重量级锁，效率并不高
    • jdk1.6版本是优化synchronized的重要版本，这一版本升级了synchronized锁，包含了偏向锁，轻量级锁，适应性自旋锁，锁消除，锁粗化等
    • 升级过程：无锁 –> 偏向锁 –> 轻量级锁 –> 重量级锁

  2. Java对象的布局

    • 对象在内存中布局是：对象头，实例数据，对齐填充
    • 对象头：内部有描述锁信息的部分，分成markworld和klasspointer
      • hotspot用instanceOopDesc来描述对象头，这个instanceOopDesc继承自OopDesc，而这个类中就有描述锁信息的字段mark word
      • 如下图
    • 实例数据：就是类中定义的成员变量
    • 对齐填充：填充为8字节的整数倍

  3. 第一个优化：偏向锁

    • 研究发现，在多数情况下，锁一般不存在多线程竞争，而由一个线程多次获得，所以偏向锁就是用来减少线程获得锁的代价
    • “偏”就是指偏向于第一个获得它的线程，会在对象头偏向锁标志设为1，并且前54位存放获取偏向锁的线程id
    • 一旦有线程要竞争，就要撤销偏向锁，变为轻量级锁。所以撤销偏向锁的性能必须小于之前节省的CAS原子操作的性能，否则就得不偿失了
    • 生成原理：
      • 第一次线程进入同步代码块，会把偏向锁标志置1，锁标志位置01，同时前54位置为线程id
      • 后面再进入时，只要比对线程id，若一致可以直接进入
    • 撤销原理：
      • 撤销偏向锁必须处于全局安全点（所有的线程都被暂停的时刻）
      • 首先暂停所有线程，包括获得偏向锁的线程，判断线程是否处于锁定状态（标志位为01）
      • 撤销，恢复到无锁（00）或轻量级锁（00）
    • 结论分析
      • 偏向锁对于单个线程性能很好，但是多个线程竞争情况下就是一个权衡利弊的问题

  4. 第二个优化：轻量级锁

    • 只有特定情况下开销比较小，所以不能替代重量级锁

    • 目的是为了在多线程交替执行同步代码块时，尽量避免重量级锁引起的性能消耗。但是如果多个线程在同一时刻进入同步代码块，就会膨胀为重量级锁，轻量级锁效果就失去了。

    • 获取轻量级锁原理

      • 先判断当前对象是否处于无锁，若是，则vm首先在当前线程的栈帧中建立一个名为锁记录（Lock Record）的空间，用于存储锁对象目前的Mark Word的复制（官方称为Displaced mark word），将对象的markword复制到lock record中，并且lock record中的owner指向当前对象
      • JVM利用CAS操作尝试将对象的mark word更新为指向lock record的指针，如果成功表示竞争到了锁，就将锁标志位变成00，执行同步操作
      • 如果失败，则判断当前对象的mark word是否指向当前线程的栈帧，如果是则表示已经持有当前对象的锁，则直接执行同步代码块。否则说明锁已被占，此时轻量级锁要膨胀为重量级锁，锁标志变成10，后面等待的线程就要阻塞

    • 撤销轻量级锁原理

      • 取出在获取轻量级锁保存在Displaced mark word的数据
      • 利用CAS操作将取出的数据替换当前对象的mark word，如果成功，则释放成功
      • 如果CAS失败，说明有其它线程尝试获取该锁，就要膨胀为重量级锁

  5. 自旋锁

    • 轻量级锁升级为重量级锁，开销会大，所以为了优化这一步，于是在轻量级锁升级过程中，加入了自旋锁优化
    • 研究发现，执行同步代码块时间是比较短的，而这个时间却需要其它线程在等待阻塞，直到被唤醒再去获取，这是不划算的。事实上只需要外面的线程多试几次就会得到。
    • 很显然，自旋锁适用于同步代码块执行时间短，同时要求硬件支持多线程并行。
    • 当然自旋锁也会消耗cpu性能，所以一般默认自旋次数的10次，10次获取不到就要阻塞了
    • 但是自旋次数是很难确定的，所以jdk1.6中引入了适应性自旋锁
    • 适应性自旋锁会根据上一次获得锁前的自旋次数来确定下一次自旋次数

  6. 锁消除

    • 如下面代码：

    • public class LockElimination {
    public static void main(String[] args) {
        contactString("aa", "bb", "cc");
    }
     
public static String contactString(String s1, String s2, String s3) {
        return new StringBuffer().append(s1).append(s2).append(s3).toString();
    }
}
     
     
@Override
@HotSpotIntrinsicCandidate
public synchronized StringBuffer append(String str) {
    toStringCache = null;
    super.append(str);
    return this;
}

    • 分析：首先普通同步方法使用的锁是this，也就是new StringBuffer

    • 而此处，三次append，很显然生成了三个StringBuffer对象，事实上没有构成竞争，但却获取和释放锁

    • 所以这里优化了锁消除，如果JIT发现尽管代码上有同步，但事实上没有竞争共享数据，就会撤销锁，进行性能优化

  7. 锁粗化

    • 如下面代码

    • StringBuffer sb = new StringBuffer();
for (int i = 0; i < 100; i++) {
    sb.append("aa");
}

    • 正常情况下，这个代码会进行100次的synchronized操作，耗费很大，所以有了锁粗化优化

    • 相当于把synchronized放在了for循环之外，只需要进行一次大的锁获取和锁释放

    • 本质就是JVM探测到一连串细小的操作都使用同一个对象加锁，于是就会把同步代码块范围放大到这一连串操作之外

  8. 平时写代码时的优化

    • 减少synchronized的范围，减少同步代码块执行时间，减少锁竞争，减少膨胀到重量级锁的情况
    • 降低synchonized锁的粒度，尽可能的将一个锁拆分为多个锁来提高并发度。最典型的例子就是HashTable和ConcurrentHashMap。再例如LinkedBlockingQueue，入队和出队使用不同的锁，是实现读写分离的一种情况
    • 读写分离：如ConcurrentHashMap，CopyOnWriteArrayList等，读取时不加锁，写入和删除时加锁
